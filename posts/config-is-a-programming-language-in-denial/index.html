Complex Configuration Is a Programming Language in Denial

date 2026-06-07 read 6 min tags #infra #guix #scheme

A few years ago, as I started my DevOps/infrastructure engineering journey, I had to learn all the tooling available in the ecosystem. At first, I felt joy writing Helm Charts, Terraform modules, Ansible playbooks. But as time went on, I found myself being more and more frustrated with minor annoyances in these tools.

A code smell is the feeling that something is wrong when reading a piece of code. It may hint at a deeper problem in how a line, a function is written, or even the architecture of the code.

The more I used these tools, the more I felt that something was wrong with them. Here is what most Helm charts look like now:

{{- if .Values.ingress.enabled }}
  rules:
    - host: {{ .Values.ingress.host | quote }}
      http:
        paths:
        {{- range .Values.ingress.paths }}
          - path: {{ .path }}
            pathType: {{ .pathType | default "Prefix" }}
        {{- end }}
{{- end }}

The templating takes more space than the code. Something is wrong.

Another smell is that lots of Helm charts use indent/nindent. Now that I look back, I wonder how we accepted at some point to template YAML files (i.e., structured data) with a text engine.

Terraform also smells. If you used a little bit Terraform, I'm sure you used something like count = var.enabled ? 1 : 0 to conditionnaly create a resource. I could add that Terraform's merge function is a shallow merge, and there's no way to do a native deep merge.

All of these tools implemented some kind of Domain Specific Language. For Helm, it's the Go template engine with a minimal standard library. For Terraform, its HCL. Other tools like Ansible and Kustomize also uses DSL: the use of when, includes and block in Ansible is an attempt at that! And Kustomize and hieradata do it with a custom YAML syntax and file structure. These all are languages in YAML costumes!

But all these DSL are intentionnally limited. The standard library they allow is fixed. And I feel constrained using them. What if we used a real programming language?

What a real language makes possible

Here's an example of what the configuration of several Kubernetes resources could look like, written in Scheme. Ignore the parentheses, the point is what real computation capabilities brings:

(with-namespace "loulou"
  (app #:name "loulou" #:image "secure.io/loulou:2.1" #:port 9000 #:replicas 2
       #:env-from '((configMap "loulou-config"))
       #:volumes  '((secret "loulou-secret" "/etc/loulou/secret"))
       #:resources "200m-*/256Mi")
  (configmap #:name "loulou-config"
             #:data '((LOG_LEVEL . "debug") (CACHE_SIZE . "256")))
  (secret    #:name "loulou-secret"
             #:data '((DB_PASSWORD . "bG91bG91LXNlY3JldA=="))))

(checksum-config)
(compliance-all "secure.io")

In the example, app is a function returning a Deployment and a Service. In Terraform, you would have to build a new module to do this abstraction, but here, it comes nearly free.

The Deployment mounts a secret at /etc/loulou/secret. I did not have to add to both the volumes and the volumeMounts sections.

Writing resource definition is annoying. As a proof of concept, the string "200m-*/256Mi" is a resource spec parsed by code into requests and limits. It manages defaults and best practices, like "memory request" = "memory limit".

Apply systemic changes to all the resources

To be honest, the strongest part of this example is this line:

(checksum-config)

Each time I wrote a Helm chart, I manually add an annotation with the checksum of all the Configmap and Secrets it depends on. This ensures that the Pods of the Deployment are restarted whenever the Secrets or Configmap are modified.

With this line, instead of having to add the checksum computation manually for each Deployment by passing explicitely the Secrets and Configmap it depends on, all workload resources are automatically patched!

After every resource is built, the function checksum-config walks the whole set, finds each Deployment, resolves the ConfigMaps and Secrets that Deployment mounts, hashes their contents, and writes the digest onto the pod template as an annotation. Change a ConfigMap and the annotation changes and the pods roll — config-driven rollouts across every workload in the repo. And the function also tracks references to a ConfigMap that doesn't exist, and reports its findings.

Compliance for free

Using code, it's also very easy to check compliance rules. For example you can ensure that:

  • every workload sets requests
  • there's no privileged containers
  • images only comes from a specific registry

Construction, derivation, validation — one substrate, one in-memory value, before anything is applied. Helm does the first, badly. Kustomize does it by merging. A policy engine does the third, separately, in Rego, after the fact. Nothing else does all three in one place, because each is computation — the thing "config is data, not code" was meant to forbid.

Best practices

Here are some best practices I found while using real programming languages:

  1. Always review the rendered diff, and not just the configuration code. If there's a bug in the code, it can become a systemic bug and affect all your resources. Let's say the parsing of the resource-spec string I mentioned earlier fails and inverts cpu and memory requests. This could be dramatic.
  2. This also means that the library part of your code needs unit tests to catch these kinds of bugs.
  3. The compliance rules in the rendering engine is shift-left. It does not replace compliance tools that run in your clusters.
  4. Your rendering code has to be deterministic. You now have freedom, you need to use it carefully.

Presenting Hexol

Hexol is a rendering engine written in Guile Scheme. I chose Guile because of one feature shared by all LISP implementations: Homoiconicity, the property of a language to manipulate a code written in the same language as data.

A Hexol inventory is a Guile file that returns a list of Operations. Each Operation is a function that takes a State as input, manipulates it, and returns the new state. By chaining all the operations (in functional programming the term is "folding"), the program actually generates the inventory.

That's it. The core of the library provides Operations for basic tasks (deep-merge, delete, copy, move, etc), and the rest of the library provides helper functions for a few specific types of inventories (Terraform, Kubernetes, Ansible, and even Ledger and Database Schemas).

This blog post is not about Hexol, so I'll leave it at that.

If you struggle with complex inventories, many manifests and fight tools like Helm and Terraform, please consider using real programming languages to generate your inventories.
